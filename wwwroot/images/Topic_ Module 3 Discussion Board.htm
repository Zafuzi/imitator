            6to4 is a Band-Aid fix, and not a terrible one. ZDnet gives an alarmist's excuse for dismissing it. Packet encapsulation is common place (GRE, ESP, etc), and the 6to4 tunneling technique does nothing to increase or decrease the security of the IPv4/6 protocol. Encapsulated packet headers should be verified against the source 6to4 IPv6, the 6to4 prefix (2002::/16), and its associated router IPv4 address as per RFC3496. A properly configured router/IDS will mitigate the majority of the aforementioned security concerns with the 6to4 technique.

                  1. TCP and UDP are vastly different, and each is suited to the particular needs of the services using them. When using TCP, a verified connection must be made through a three-way hand shake (SYN - SYN/ACK - ACK) before any data can be transferred. The host makes a request(SYN), the server then responds (SYN/ACK), and the host responds back (ACK). Once this handshake is complete, the bilateral transfer of data can commence. The TCP protocol is used in applications where reliability is necessary, but speed is not, as TCP is inherently slower when compared to UDP. TCP also performs error-checking and flow control, which are more time-consuming, but ensure that all packets are transmitted and received in the exact same order they were sent, and network congestion is handled efficiently at the transfer layer. By stark contrast, UDP is fast, unreliable, and careless. This protocol does not check to ensure the receiver is listening, nor does it offer any sort of error resolution. Erroneous packets are simply tossed, and anything not picked up by the receiving host is lost. Without acknowledgement, re-transmission, or, timeout, UDP does not make for a reliable protocol. However, because UDP is fast and light-weight, it is useful for gaming or query applications where speed counts more than integrity or reliability. 2. Many of the components worth noting have already been actualized by IPv6, including a larger maximum payload size per packet, automatic address detection (meaning two devices won't be assigned the same IP on the same network), and a simpler, more efficient header structure for faster processing at the data-link layer.

                                      TCP is the workhorse of data transmission. To make sure communication can even happen, TCP uses a three-way handshake to establish a connection. Next it uses checksum to determine if the connection is clear enough for communication. Flow control then sets a rate of transmission between the computer doing the sending and the computer doing the receiving. Just as a farmer once needed to find a steady pace at which to maintain his workhorse to keep his furrows straight, the communicating computers work out the right pace to maintain the steady flow of data. Because there will be confirmation that the data was sent over a clear channel and was received, TCP is a reliable way to transmit data. UDP doesn’t have the checks and balances that TCP does. With the UDP, the data streams one way and if it doesn’t make it all in one piece, it’s up to the user to request the information be sent again. UDP is commonly used for streaming music and videos from companies like Spotify and Netflix. With UDP, you might receive less data than was sent, resulting in an unclear picture when you’re watching Poirot. Or perhaps the data is streaming in more slowly than the speed of the show. Either way, with UDP there’s no monitoring the stream of data in the same way there is with TCP. It kind of works out, though, since it would be weird if a show or some music started to play at the rate the data was being transmitted. If we could add NDP to IPv4, instead of relying only on the less reliable neighbor discovery process as managed by ARP and ICMP, data would flow a lot more reliably more often. I can’t help wondering if this isn’t being upgraded because the powers-that-be want to phase out IPv4 and push people toward IPv6.

                                                                                                                                                                              1. TCP and UDP are the two types of IP traffic. TCP is often used in scenarios where connection reliability is important, whereas UDP is used more frequently when latency is a greater issue. Because UDP is a connectionless protocol, meaning that it can send information without connecting to the recipient or waiting for a callback, it is commonly used in VoIP calls, some gaming instances, and DNS resolving. TDP on the other hand is a connection based protocol and must have a solid connection to each node otherwise the connection will be terminated. For this reason HTTP, FTP, and SMTP utilize TCP because it is more reliable than UDP and they are all protocols that rely on a solid connection. This also means that TCP is slower than UDP and less useful in situations that require quick communications like video calling.   2.
